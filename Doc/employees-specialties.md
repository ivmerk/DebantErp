## Обзор

Экран «Работники» открывается из хаба **Данные** (`/workspace`). По умолчанию
данные показываются **только для чтения**; добавление и изменение появляются
после нажатия кнопки **«Изменить»**.

Возможности:
- Список работников (сортировка по фамилии, пагинация по 20).
- Создание работника со всеми обязательными полями.
- Редактирование ФИО и деталей (телефон, почта, ИНН, адрес, дата рождения, пол).
- Мягкое удаление работника (`is_actual = false`).
- Назначение, показ и снятие специальностей с датой начала.
- Экран трудозатрат работника за период (`/workspace/employees/{id}/labor-costs`):
  таблица «дата заказа · заказ · операция · ставка · количество · сумма» с итогом
  и фильтром по датам (по умолчанию — текущий месяц). Период фильтруется по
  **дате заказа** (`orders.created_at`) — отдельной даты у трудозатраты нет.
  Использует `IOrderLaborCost.GetByEmployee` (partial-индекс по `employee_id`);
  ставка резолвится по конкретной версии расценки.

## Архитектура

Серверный рендеринг ASP.NET Core MVC (Razor + формы), PostgreSQL через Dapper.

Маршрут раздела: `[Route("workspace/employees")]`. URL не меняются при доработках.

## Этап 1. Контроллер и сервис работников

Сервис `IEmployee` / `Employee` и DAL уже существовали; добавлен только
контроллер. `Employee.Get()` больше не бросает исключение на пустом списке —
возвращает пустой список (как у специальностей).

## Этап 2. Элемент на дашборде

Плитка «Работники» в `Views/Workspace/Index.cshtml` ведёт на
`EmployeesController.Index`. Экшен отдаёт список с фильтром `is_actual`,
сортировкой по фамилии→имени и пагинацией.

## Этап 3. Редактирование деталей работника

Форма редактирования строки раскрывает блок «Детали» (Bootstrap collapse).
`Employee.Update` после обновления ФИО вызывает
`IEmployeeDetails.UpdateEmployeeDetails`, обновляя ИНН, адрес, почту, телефон,
дату рождения и пол. Исправлены два бага DAL: `Update` не писал нужные поля,
`Get` обращался к неверной таблице.

## Этап 4. Разбиение контроллеров (по сущностям)

`WorkspaceController` разделён на контроллеры по сущностям, чтобы файл не рос:
- `WorkspaceController` — только хаб (`Index`) и заглушки `Rates`/`Orders`.
- `EmployeesController` — раздел «Работники».
- `SpecialtiesController` — раздел «Специальности».
- `WorkspaceBaseController` — общий предок с `PageSize`.

Представления переехали в `Views/Employees` и `Views/Specialties` по конвенции
(`Index` → `Views/<Controller>/Index.cshtml`).

## Этап 5. Режим просмотра / редактирования

Режим управляется серверным флагом `?edit=true` (модель `Edit`), а не CSS — в
просмотре форм и кнопок правки **физически нет** в разметке.

- Кнопка **«Изменить»** → `?edit=true`; **«Готово»** → обычный URL.
- В просмотре поля рендерятся как `form-control-plaintext` + `readonly`.
- После создания/правки/удаления редирект сохраняет режим (`edit = true`).
- Пагинация сохраняет режим (`asp-route-edit`).

## Этап 6. Специальности работника

Через `EmployeeSpecialtyAssignment` у работника показываются и назначаются
специальности.

- **Показ** (оба режима): чипы вида «Название · с ГГГГ-ММ-ДД». Название и дата —
  тёмным шрифтом (`text-dark`).
- **Назначение** (режим правки): форма «выбор специальности + дата начала» →
  `POST /workspace/employees/{id}/specialties`.
- **Снятие** (режим правки): крестик на чипе →
  `POST /workspace/employees/{id}/specialties/{assignmentId}/delete` (мягкое,
  `is_actual = false`).
- **Реактивация**: повторное назначение ранее снятой специальности обновляет
  существующую строку (обходит `unique(employee_id, specialty_id)`), а не
  создаёт дубликат.

В `EmployeesController.Index` имена специальностей резолвятся через словарь
`id → name`; список активных специальностей передаётся для выпадающего меню.

## Этап 7. Правило даты начала специальности

**Дату начала новой специальности нельзя задать раньше самой поздней `date_from`
среди уже назначенных работнику специальностей.**

Реализация на двух уровнях:
- **Сервер** (`AssignSpecialty`): сравнивает дату с `Max(date_from)` активных
  назначений работника; при нарушении — отказ с сообщением
  *«Дата начала не может быть раньше последней назначенной специальности
  (ГГГГ-ММ-ДД)»*.
- **Страница**: у поля даты атрибут `min` = последняя `date_from` работника, так
  что более раннюю дату нельзя выбрать в браузере.

Граничные случаи: равная дата разрешена (запрещается строго более ранняя); у
работника без назначений ограничения нет.

## Запуск и проверка

```bash
# БД (из каталога DebantErp/)
docker-compose -f docker-compose.dev.yml up -d

# Сборка и запуск (из корня / каталога DebantErp/)
dotnet build DebantErp.sln
cd DebantErp && dotnet run            # http://localhost:5010
```

Вход (сид-админ): `s2P7D@example.com` / `123456`.

> ⚠️ При запуске убедитесь, что собрана **актуальная** версия (`dotnet build`),
> иначе можно тестировать старую сборку и видеть устаревшее поведение.

Сценарий проверки правила даты:
1. Открыть «Работники» → «Изменить».
2. У работника с уже назначенной специальностью (например, дата 2026-05-08)
   попробовать назначить новую с более ранней датой (2026-03-01) — **отказ**.
3. Назначить с более поздней датой — **принимается**.

Освободить порт, если занят прошлым запуском:
```bash
lsof -ti tcp:5010 | xargs kill
```

## База данных

Схема создаётся **вручную** скриптами `Db/001_*.sql` … `Db/008_*.sql` по порядку
(раннера миграций нет). Данные засеваются `MockDataSeeder` при старте, если
таблицы пусты.

Таблица назначений (`Db/003_create_specialties.sql`):

```sql
create table if not exists employee_specialty_assignments (
    id serial primary key,
    employee_id int references employees(id) on delete cascade,
    specialty_id int references specialties(id) on delete cascade,
    date_from date not null,
    is_actual boolean default true,
    created_at timestamptz default now(),
    updated_at timestamptz default now(),
    unique(employee_id, specialty_id)
);
```

### Важно: опечатка в старых БД
Старые dev-базы были созданы с таблицей `employee_speciality_assignments` и
колонкой `speciality_id` (лишняя «i»). Код и скрипты используют правильное
`employee_specialty_assignments` / `specialty_id`. Dev-база пересоздаётся из
скриптов `001`–`008` начисто:

```sql
DROP SCHEMA public CASCADE;
CREATE SCHEMA public;
-- затем применить Db/001..008 по порядку
```

На других окружениях, где есть опечатка, нужно либо пересоздать схему, либо
переименовать таблицу и колонку (`ALTER TABLE … RENAME …`).

## Известные ограничения

- **Снятие + повторное назначение с более ранней датой.** После снятия
  специальности отсчёт минимальной даты идёт по оставшимся активным назначениям,
  поэтому повторно назначить специальность с датой ниже её прежней — возможно.
  Если такое нужно запрещать — правило отсчёта следует расширить (учитывать в том
  числе снятые назначения или историческую максимальную дату).
- **Редактирование даты существующего назначения** отдельной формы не имеет —
  меняется только повторным назначением той же специальности (реактивация).
